# Резервне копіювання (backups)

Щоденний бекап бази даних і завантажених документів CRM.

## Що бекапиться

- **База даних** (PostgreSQL) — клієнти, справи, документи, фінанси, користувачі, аудит.
- **Файли документів** (том `media`) — завантажені PDF/DOCX тощо.

Обидва пишуться у `/var/backups/crm/` на сервері:

```
db-2026-06-28-033001.dump      # дамп бази (pg_dump -Fc)
media-2026-06-28-033001.tar.gz # архів файлів документів
backup.log                     # журнал запусків
```

## Як це працює

- Скрипт: `deploy/backup.sh`. Бере дамп бази з контейнера `db` і архівує том `media`.
- Розклад: cron-файл `/etc/cron.d/crm-backup` запускає його **щодня о 03:30**.
- Зберігання: останні **30 днів**, старіші копії видаляються автоматично.
- Облікові дані бази в скрипті не зберігаються — береться з оточення контейнера.

Запустити вручну (на сервері):

```bash
/opt/crm/deploy/backup.sh
```

## Відновлення бази

```bash
/opt/crm/deploy/restore.sh /var/backups/crm/db-РРРР-ММ-ДД-ччххсс.dump
```

Скрипт спитає підтвердження, перезапише поточну базу і перезапустить `web`.

## Відновлення файлів документів

```bash
docker run --rm -i -v advokat-crm_media:/media postgres:16-alpine \
  sh -c 'cd /media && tar xzf -' < /var/backups/crm/media-РРРР-ММ-ДД-ччххсс.tar.gz
```

## Off-site (копія поза сервером) — рекомендовано

Локальні бекапи рятують від помилкового видалення, поганої міграції чи збою бази,
але **не** від втрати самого сервера. Для повного захисту потрібна копія в іншому місці.
Варіанти (потрібно обрати призначення й доступи):

- хмарне сховище (Backblaze B2 / S3) — щодня заливати `/var/backups/crm/` через `rclone`;
- інший сервер чи робоча машина — `rsync`/`scp` за розкладом;
- ⚠️ дані містять адвокатську таємницю → off-site копії бажано шифрувати (наприклад `age`/`gpg`).
