## Общий вердикт

Архитектура здоровая: единый источник данных (`state`), экраны — чистые функции рендера,
аккуратный выбор источника (snapshot → API → демо), грамотный скоупинг доступа к делам на backend.
Базовый юридический контур (клиент → дело → задачи → документы → финансы) реально работает через Django API.

Дефекты делятся на 3 категории:
1. **Безопасность / авторизация** — фактически отключена.
2. **Целостность данных** — есть сценарии крашей и осиротевших записей.
3. **«Честность» данных** — рядом с реальными цифрами показываются захардкоженные/выдуманные.

## 🔴 Критичные проблемы

### Безопасность (backend)
- **Авторизация декоративна.** `backend/config/api.py:2205` — `current_demo_user()` при запросе без сессии
  возвращает первого админа. Все проверки прав построены поверх неё, а не поверх залогиненного пользователя.
  Любой анонимный запрос исполняется от имени админа. Нет `@login_required`, нет middleware на сессию.
- Почти все эндпоинты `@csrf_exempt` + `CORS: *` с `Allow-Credentials: true`.
- **SSRF** в ONLYOFFICE-колбэке: `api.py:2705` сервер скачивает произвольный URL из тела запроса без авторизации.

### Целостность данных
- **Удаление дела с финансами падает или рвёт связи.** `api.py:2562` — чистятся только задачи/события,
  а `Invoice`/`Payment` имеют `on_delete=PROTECT` → `ProtectedError` (500). Документ-счёт (`CASCADE`) при этом исчезает,
  оставляя `Invoice` висеть.
- **Удаление одной справы во фронте не чистит события календаря.** `frontend/js/dialogs.js:191-205` → осиротевшие события.
  Массовое удаление это делает (`frontend/js/screens/cases.js:1109`) — поведение рассогласовано.
- **Создание события крашится при пустом списке дел.** `frontend/js/forms/events.js:27-41` —
  `caseItem` становится `undefined`, `caseItem.id` → `TypeError`.
- **Неверная финансовая арифметика.** `api.py:1028` — при оплате доход «подтягивается» до суммы платежей,
  поэтому переплата невозможна, а долг всегда обнуляется.
- **Коллизия id справ.** `frontend/js/forms/cases.js:44` — новый номер = `length + 1112`. После удаления счётчик
  откатывается → возможен дубль первичного ключа, на котором держатся все связи.

### Логика приложения (frontend)
- **`showToast` — пустая заглушка.** `frontend/app.js:397` функция только очищает контейнер и ничего не показывает,
  при этом вызывается 181 раз. Все ошибки и подтверждения невидимы пользователю.
- **Snapshot-режим теряет ручные данные.** `persistSnapshotState` вызывается только при удалении
  (`frontend/js/dialogs.js:321`). Ни одна форма создания/редактирования не сохраняет snapshot.
- **Права для API без сессии = всё разрешено.** `frontend/app.js:350` — пустые права уходят в fallback
  `defaultPermissions`, где всё `true`.
- **Зарплата не персистится.** `state.salaryRows` — локальный state, нет модели/эндпоинта. То же с `state.documentArchiveFolders`.

### «Фейк под видом реальных данных»
- **Дашборд:** все тренды «+12%», «+100%» захардкожены (`frontend/js/screens/dashboard.js:282`).
- **Аналітика:** линейный график «Динаміка справ» сломан — лейблы зашиты на май, а демо-даты сдвинуты на сегодня
  (`frontend/js/derived-data.js:251`); финансовые столбцы захардкожены (`frontend/js/screens/analytics.js:248`); тренды выдуманы.
- **OSINT:** date-picker ничего не фильтрует; граф связей — статичный SVG независимо от дела; реестры — фиктивные.
- **AI:** «Всього запитів» = выдуманная формула; ответы чата — if-else по ключевым словам, без LLM.
- **Розсилка:** коэффициенты доставки и числа получателей выдуманы; фильтры сегмента ни на что не влияют.
- **Планер:** продуктивность подделана формулой (`frontend/js/screens/planner.js:104`).
- **Документи:** две расходящиеся копии `inferCaseDocumentFolder` (`frontend/js/screens/documents.js:8`
  vs `frontend/js/forms/documents.js:8`) дают разную автораскладку; функции «переместить документ» нет.

## 🟡 Средние проблемы

- Много кнопок-пустышек (тост вместо действия): «Синхронізувати планер», «Згенерувати план», «Оновити OSINT»,
  «Навчити на матеріалах AI», селекты переключения графиков в аналитике.
- Backend: нет обработки битого JSON (`parse_body` → 500 вместо 400), нет валидации тел запросов (минуют `choices`).
- Финансы: «баланс на рахунках» = прибыль за период, «готівка» захардкожена в 0; проценты прострочки по индексу строки.
- Запрещённый контент (финансы, список пользователей) рендерится в DOM целиком — скрываются только кнопки.
- Поле `consent` клиента записывается, но не читается при выборе получателей рассылки.
- Утечка обработчиков в `bindViewLinks` (`frontend/app.js:771`) — нет guard на повторное навешивание.
- Смысловые подмены: «дедлайн» подписан как «завершені справи», «тип дела + право» = «топ практик».
